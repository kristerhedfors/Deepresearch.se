# feature-maintenance — route regressions back to the fix's author

Some features regress repeatedly (the in-browser Linux **sandbox** above all:
boot hangs, "sandbox not ready", stream-stalled). A fix is **not done at merge** —
it needs an owner who is pinged when it breaks and re-resumes to fix it. This
skill is how the watcher/merger keeps features working by routing breakage back
to the worker who wrote the fix, instead of silently re-fixing.

## The back-channel

- Every fix PR is authored by a **worker session**, and each worker **stays
  subscribed to its own PR** (`subscribe_pr_activity`). (Owner set this up
  2026-07-14: "workers watch every PR".)
- **A GitHub comment on a PR wakes that PR's author-worker** — it auto-resumes
  with full context and kicks off a fix. So: *to reach the author of a fix,
  comment on their PR* (`mcp__github__add_issue_comment`).
- The author's `Claude-Session:` trailer / "Generated by Claude Code" footer in
  the PR body identifies the worker; `docs/MAINTENANCE-OWNERS.md` records who
  owns what.

## The loop (per watch tick, alongside merging)

1. **Detect.** Sweep shipped features for regressions — `scripts/chatlogs`
   keyword search for the failure signatures in `docs/MAINTENANCE-OWNERS.md`
   (sandbox: `sandbox not ready`, `stream stalled`, `sandbox.boot_stalled`,
   `sandbox.exec_timeout`, high `client_diag.fs.ms`), plus live probes / user
   reports. See **chat-logs**, **live-verify**, **bugreport-bugfix**.
2. **Attribute.** Find the owning PR: the most recent merged PR touching the
   broken files (confirm against the registry). Its trailer names the worker.
3. **Route back — don't silently fix.** Comment on that PR with an **actionable
   regression report**:
   - the symptom in one line,
   - the `chat_logs` id + the telling `client_diag` counters,
   - the **verbatim** user message + settings that reproduce it,
   - which prior fix it regressed (PR #, commit),
   - what "fixed" looks like (the observable success condition).
   That wakes the worker; it fixes and opens a follow-up PR.
4. **Merge the follow-up** the normal way (watch/merge loop). The follow-up PR
   becomes the new owner → **update `docs/MAINTENANCE-OWNERS.md` in that pass.**
5. **Fallback.** If the owning PR is closed/stale or the worker doesn't respond,
   fix it yourself WITH a regression test (the **feedback-loop** discipline) and
   record it in the registry.

## Standing sandbox maintenance owner

The sandbox is the reason this exists. Keep one worker **in the loop**
(subscribed) to uphold it until the boot is reliably green on a real iOS PWA. The
recurring root cause is the CheerpX `/workspace` IndexedDB mount stalling on iOS
WebKit / Firefox iOS (~11–27 s vs ~0.8 s Safari-tab), which cascades to
"sandbox not ready" / stream-stall. Loop-level fixes (#34 exec timeout, #37
teardown stop) make it fail SOFT; the ROOT fix (mount model) is live-on-device
work — that's the Playwright worker's standing charge. Ping that PR on every
recurrence; don't consider it closed until the sandbox boots green on device.

## Guardrails

- **One report per distinct regression.** Don't spam the PR — dedupe against what
  you already reported (a repeat signature with the same root cause is a nudge,
  not a new report).
- **Precise, not vague.** "sandbox broke" is useless; a `chat_logs` id + verbatim
  repro + counters is what lets the worker fix it in one resume.
- **Never leave a regression unrouted.** If you detect it, either it goes to an
  owner or you fix it — never observed-and-dropped.
- Keep the registry honest: newest fix PR for a subsystem = its owner.
