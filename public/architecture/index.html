One product, two answers to the same question

DeepResearch.se is a research project on the privacy capabilities of LLM applications, and this pair is the proof. It ships the same deep-research assistant twice, as a deliberate pair: DeepResearch.Se/cure, the public client-side tier, where your browser is the research engine and the server is deliberately absent from every data path — and DeepResearch.Se/rver, the signed-in tier, where a Cloudflare Worker orchestrates the research. Same floating glass UI (Se/cure in khaki), same five-phase pipeline shape, same no-function-calling determinism — and opposite answers to the question "where is the user's data allowed to exist?".

The short version of the trade this page visualizes: Se/cure buys privacy structurally — the server cannot log, leak, or be compelled to reveal what never reaches it — and pays for that with the capabilities only a server can hold keys for. Se/rver buys capability — live web search, maps, host intelligence, attachments, accounts — by having a server, and protects the data that server handles with encryption and policy.

The two data paths

Privacy, paired

The same concerns, answered by each architecture. The pattern to notice: on DeepResearch.Se/cure each row is a consequence of the structure — there is no server-side behavior to trust, because there is no server-side behavior; on DeepResearch.Se/rver each row is a promise the code keeps.

DeepResearch.Se/cure DeepResearch.Se/rver
Identity No accounts at all. Nothing to sign in to; nothing to attribute a question to. Google sign-in, invite-only approval, quotas per account.
Conversations at rest One sealed archive in the browser's own storage: chats, the RAG index (chunk text and vectors), and your provider API keys, AES-256-GCM under a secret only you hold. Stricter than Se/rver's RAG exception — even indexed material rests as ciphertext. Ciphertext in the browser (IndexedDB, AES-GCM) and — always, this is Se/rver — the same ciphertext in R2. Readable exceptions, because retrieval needs plaintext: RAG-indexed material and project chats. The key is derived server-side, with no user-held key to manage — so this is not perfectly private: the running service can derive that key and read Se/rver history. That is the honest cost of cloud storage without external key management (and the whole stack is open-source, so you can verify there's no hidden path — and so can an attacker; it's still ordinary, vulnerable software). The copies even the operator can't read: a project's secret-keyed vault, or Se/cure — nothing server-side at all.
Interaction log Structurally impossible. The question and answer travel browser → provider on your key; the server never carries either, so there is nothing it could log. A full-visibility chat log (complete question, answer, research metadata) in D1 — an explicit product decision (2026-07-08) that powers the agentic debugging and feedback loops. Suppressed for any request carrying the incognito flag — the API promise the ghost feature made stays honored.
API keys Your own keys, pasted once, living inside the sealed state in your browser. Each key is sent only to its own provider, straight from the browser. Server-side secrets (Berget, Anthropic, OpenAI, Exa, Maps, Shodan). Users never handle a key; keys never appear in any log.
What third parties see The provider you chose sees your questions (that's inherent to using any model API) — under your account and data-processing terms with them, not ours. The minimum for the job: a search query, a coordinate, a hostname — never the conversation, a filename, or the account identity.
The strictest tier The whole tier is that property. The vault's crypto core (vault-core.js) is literally the foundation Se/cure's sealed state is built on. The secret-keyed project vault: a whole project archived server-side as ciphertext under a user-held secret the server never sees and cannot derive. One feature with the structural property.
In one line Privacy by structure — no promise needed. Privacy by encryption and policy — auditable promises.

Capabilities, paired

What each side of the trade buys. Rows where Se/cure matches Se/rver are the deliberate ports; rows where it can't are exactly the capabilities that require server-held keys or a server-side data path.

DeepResearch.Se/cure DeepResearch.Se/rver
Research pipeline The same five-phase shape, ported to the browser — with the search wave replaced by an offline knowledge harvest. Same determinism, same fail-soft invariants, verified end to end in tests. Triage → search → gap check → synthesis → validation, orchestrated by the Worker. Deterministic, no function calling, every helper phase fail-soft.
Sources No web search (no server to hold the Exa key). The harvest phase pools the model's own knowledge, and the prompts force honesty about that limit. Live Exa web search waves with a numbered, domain-diverse source registry; Hugging Face Hub joins as citable sources when asked.
Models The CORS-capable providers only — OpenAI, Groq, Berget — because the browser must be able to call them directly. Split routing survives: JSON phases run on each provider's fixed cheap JSON model. The full Berget catalog plus key-gated Claude and GPT answer models; JSON planning phases always on the fixed reliable model (split routing).
Documents & RAG Project & chat RAG, browser-direct embeddings on your key (OpenAI small/512-dim by design; Berget too; a Groq-only session simply runs without RAG — fail-soft). Attachments (pdf/docx/images with metadata extraction), project files, chat RAG — embeddings via the Worker's /api/embed proxy.
Enrichments None. These are exactly the capabilities that exist only because a server holds the keys and takes the minimal-disclosure role. Google Maps / Street View scenes, Shodan host intelligence, reverse geocoding, the image deck — all opt-in knobs, all server-keyed.
Accounts & continuity Browser-local projects with save-as-project and /my/project-<hash> deep links; published research replays open in place and continue on your own keys. No accounts to sync. Quotas, usage windows, always-on cloud sync (ciphertext), cross-device projects, the message center, feedback mode, quizzes, games.
Execution sandbox Shared — and identical. The same x86 Linux VM boots in the browser tab on both tiers (next section). The only difference is who picks the next command: on DeepResearch.Se/cure even that decision happens client-side on your key; on DeepResearch.Se/rver the Worker's reliable model does (/api/bash/step). Execution never leaves the browser on either tier.

The shared engine room: a real Linux inside the browser tab

The experimental execution sandbox (the bash-lite agent, default off) is the clearest expression of the whole architecture: even on the server tier, the server never runs a shell. A CheerpX x86 emulator, compiled to WebAssembly, boots a stock Debian Linux entirely inside the page; the agentic loop feeds it commands through a fenced-block convention (no function calling, like everything else here), and the resulting transcript feeds synthesis as ground truth.

Which Linux, exactly — and why "big" is fine

The disk is the public WebVM image debian_large_20230522 — a stock Debian, about 4.7 GiB of ext2, streamed over WebSocket from disks.webvm.io. That sounds like the wrong image for a browser until you see how it's read (a deliberate keep-as-is decision, reviewed 2026-07-12):

  • Only touched blocks download. The disk streams lazily, block by block; a session that runs python3 and jq fetches those tools' blocks and nothing else.
  • Fetched blocks are cached in IndexedDB, so repeat sessions boot warm.
  • User data lives on separate volumes — the /workspace and per-project overlays are their own IndexedDB devices, so the big base image can be reset or re-streamed without touching your files.
  • The stock image is the capability. A full Debian means the agent finds real tools (python, coreutils, jq…) instead of a stripped busybox — and the transcript quality depends on exactly that.
  • The expensive step is avoidable anyway: the host-command design means sessions that never need the VM never boot it at all.

Honest status: this is a borrowed stock image made economical by streaming and caching, not an image tuned for us — kept deliberately, because the boot-avoidance and lazy-block economics beat an image diet, and a smaller disk would cost preinstalled tools. The CheerpX runtime and the public disk service carry their own licensing terms to review before this graduates from experimental.

And the pairing holds here too: the sandbox gives DeepResearch.Se/cure a capability that would otherwise be impossible (a no-server tier still gets code execution, because execution never needed the server in the first place), and it gives DeepResearch.Se/rver a capability without a privacy cost (your files mount into a VM that lives and dies in your tab — the server only ever suggests the next command).

Why a pair, and not a setting

The two tiers share their UI, their pipeline shape, their crypto core (vault-core.js), and their sandbox — but they are separate architectures, not a toggle. A privacy property you can switch off is a policy; a privacy property the structure enforces is a guarantee. Keeping both tiers whole means every user gets to pick which trade they're making, and the ghost button in the main app is the door between them: it doesn't flip a flag anymore, it walks you to DeepResearch.Se/cure.
