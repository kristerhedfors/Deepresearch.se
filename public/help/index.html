How DeepResearch.se works

← Back to the app

What this site does

You ask a question; the site plans web searches, runs them, reads the results, and writes an answer with numbered citations — like a research assistant on a deadline you choose. The AI models run at Berget.ai (EU-hosted) and web searches go through Exa.

Privacy in one line: your chat history lives in your browser, encrypted (see "Chat history" below). To improve the service, the server keeps a log of completed research interactions — the question, the answer, and the research steps. For chats that should leave no trace on this server at all, the ghost button (upper right) opens ghost mode — deepresearch.se/cure, the client-side twin of this app where the server is never in the data path (your own API key, browser-local storage).

The header

  • History (clock icon, left) — opens the chat history panel: see "Chat history" below. Hidden entirely if this server hasn't set up encrypted history.
  • New chat (pencil icon) — saves the current conversation (if it has one) and starts a fresh one on screen. The old one isn't erased — find it again in the History panel, or delete it there if you're done with it.
  • Model selector (middle) — picks which AI model answers. Models that are down for maintenance show as greyed out and come back automatically. Your choice is remembered in this browser.
  • Account button (person icon, right) — opens your usage panel: see below.

Chat history — encrypted, and yours alone

Every conversation you have is saved automatically, right here in your browser, so you can pick it back up later — the History panel (clock icon in the header) lists them by their first question, most recent first. Click one to reopen it (this also restores the model, time target, and web-search setting it was sent with); the pencil renames it, the trash bin deletes it.

Where it actually lives: in this browser's IndexedDB (the modern, larger-capacity successor to localStorage). The server never holds a readable copy of your saved history; separate from history, completed exchanges go to the server's interaction log unless the conversation was started incognito (see "Privacy, in full" below).

How the encryption works

Every conversation is encrypted (AES-256-GCM) before it touches your browser's storage. The key comes from the server, derived per-account from a secret it holds — but that key is fetched fresh each time you open the app and kept only in memory for that page load; it is never written to localStorage, IndexedDB, or any other disk-backed storage. This split is deliberate, and gives two separate guarantees instead of one:

  • If your browser's storage is examined at rest — a stolen or cloned device, a forensic disk image — the attacker finds only ciphertext. The key that would unlock it was never written down in that browser to begin with.
  • If our server is ever compromised, the attacker could recover the secret the key derives from and compute your key — but would still have no ciphertext to decrypt with it, because your conversations never leave your browser in the first place.
Honest limitation: only the combination of both — someone who has compromised the server (able to derive your key) and has separately obtained your browser's stored data — could decrypt your history. That is a materially higher bar than either alone, which is the actual promise here; we're not claiming a single compromise is impossible to build into a bigger attack, only that it isn't sufficient by itself.

What this means day to day

  • It's this browser only. History doesn't sync between devices or browsers — sign in on your phone and it starts with no history, independent of your laptop's. That's the same isolation that keeps each copy's key exposure independent.
  • Clearing your browser's site data deletes it for good. There is no server-side backup to restore from — that's the same fact that makes a server compromise alone insufficient.
  • If this server hasn't configured the feature, the History button simply doesn't appear — conversations still work, they just don't persist across a reload, the previous behavior.

The exception: chats inside a project

A chat that belongs to a project is indexed for retrieval, the same way project documents and notes are — so your other chats in that project can automatically draw on what was worked out in it. The index keeps growing as the conversation does: each new exchange is added to it. Because a search index needs readable text, indexed material is stored readable, not encrypted — and that includes project chats themselves, in this browser and (when cloud storage is on) in the cloud copy. Chats outside projects are not indexed and stay encrypted exactly as described above. If a conversation shouldn't be readable at rest — or retrievable from your other chats — keep it outside projects, or take it to ghost mode (the ghost button → deepresearch.se/cure, where nothing reaches this server at all).

Backing up or moving a project with a secret

A project — even one kept out of the cloud — can be stored as one encrypted archive keyed by a secret: open the project and press Store next to "Encrypted copy, keyed by a secret". Your browser packs the project's chats, files, and search index, encrypts everything locally with a freshly generated secret (DR1-XXXX-…, 160 random bits), and uploads only the unreadable result. Unlike regular cloud storage, the server never holds anything readable — not the project's name, not its files, not even a key it could derive: the secret shown to you (once — copy it!) is the only way to find and decrypt the copy. Use 🔑 Load project from secret in the History panel on any of your devices to bring the project back. Storing again replaces the copy under a new secret, and the old secret stops working; losing the secret loses the copy — there is deliberately no recovery.

The input area

  • Text field — type your question. Enter inserts a line break; nothing is sent until you tap the arrow. The field grows up to about six lines.
  • Paperclip — attach images or documents (pdf, docx, md, txt). Each attachment appears as a small card above the bottom edge of the pane, with a white ✕ to remove it. Documents are read in your browser — their extracted text travels inside your message; the files themselves are never uploaded anywhere. Images need a vision-capable model (you'll be offered a switch) and are downscaled in your browser first.
  • Attachments can carry more than their visible content — photos often embed where and when they were taken and what device took them; Word documents can carry the author's name, edit history, and reviewer comments, sometimes including text that was deleted but is still physically present in the file. Since this is often exactly what a research question is about ("where was this photo taken", "what did this document originally say"), it's extracted along with the visible content and sent as part of your message. Whenever anything is found, the attachment's card shows a small badge before you send — a plain "ℹ️ metadata included" for routine details, or a distinct "📍 location data included" (a photo's GPS coordinates) / "⚠️ tracked changes included" (a document's unaccepted deletions) for the two cases worth a second look. Remove the attachment if you'd rather not include what the badge is telling you about. A photo's raw coordinates are also resolved server-side into an actual place name (via OpenStreetMap Nominatim — only the coordinates are sent, nothing else) since a place name is far more useful for research than bare numbers.
  • Spiderweb knob — the web-search switch, with a spiderweb inside the knob: it glows blue when search is on and turns grey when off. Tap to toggle; press and hold it for a short explanation of the switch. This is the control with the biggest privacy meaning — see the next section.
  • Slider + time — your research time target, from 15 seconds to 10 minutes. More time buys more search angles, more follow-up rounds, and a fact-checking pass; less time gets a faster, slimmer answer. The spelled-out target sits right of the slider.
  • Arrow — sends the question. This is the only way to send.

On each answer

  • Raw — toggles between rendered and plain-text view.
  • Copy — copies the answer text to your clipboard.
  • PDF — downloads the answer as a formatted report tagged DeepResearch.se, generated entirely in your browser.

Quizzes — test yourself on anything

Ask to be quizzed — "quiz me on this document", "quiz me on the French Revolution with 8 questions", "förhör mig på kapitlet" — and the answer becomes an interactive quiz: one question at a time, each with multiple-choice alternatives plus a free-text field to answer in your own words, immediate feedback with a short explanation after every answer, and a final score with a per-question recap at the end.

  • The material is up to you — the conversation so far, an attached document, your project's files and notes, or (with web search on) fresh web research on the topic you name.
  • Written answers are graded on meaning, not exact wording — synonyms, paraphrases, and another language all count. If a written answer can't be graded, it's shown as ungraded and simply doesn't count toward the score.
  • The result stays in the conversation — after finishing you can ask follow-ups like "explain the ones I got wrong", and a reopened conversation shows the finished quiz (or lets you resume an unfinished one).

Web search on / off — and what it means for privacy

On (knob blue): the AI plans a few search queries derived from your question, runs them against Exa, and answers from the sources it finds, with citations.

Off (spiderweb grey, slider dimmed): the model answers purely from its own training knowledge. No web requests are made at all.

Privacy: with web search ON, short search queries derived from your question are sent to Exa (never your full conversation) — and Exa retains those queries by default (see the next section). With web search OFF, nothing leaves except the conversation itself going to Berget.ai to generate the answer. Either way, our server stores no message content.

Sensitive topics — a two-step, semi-private workflow

Be aware: Exa is not a zero-data-retention service by default. The search queries it receives may be retained under Exa's own policies — Exa offers true zero data retention only as a paid enterprise arrangement, which this site does not currently have. The queries are short and AI-derived, never your full conversation, but on a sensitive subject even a short query reveals the topic.

If that matters for your question, split the work in two:

  1. Fetch generically. With web search on, ask a broad, impersonal question about the subject — the kind that could appear in any textbook index ("current treatment options for X", "how does Swedish law regulate Y"). The assistant searches Exa and pulls the source material into the conversation. Keep names, case details, and anything personal out of this step: it is what Exa sees.
  2. Ask specifically. Turn web search off and ask your real questions. The model answers from the sources already fetched into the conversation (plus its own knowledge) — nothing further is sent to Exa.

If the step-2 answer lacks source material, flip search back on for another generic round on the missing angle, then switch it off again and continue.

This is semi-private, not anonymous: your generic step-1 queries were still seen (and possibly retained) by Exa, and the whole conversation — including the fetched excerpts — is still processed by Berget.ai (EU) on every turn. For anything truly sensitive, keep identifying details out of the chat entirely.

Your account panel

  • Research budget — an opaque percentage of your allowance for AI-model usage in each window (last 5 hours, today, this week, this month). When a bar reaches 100% you'll be asked to wait until the window shown next to it resets.
  • Web searches — how many live searches you've used of your allowance per window.
  • Sign out — ends your session on this device. You stay signed in for a year otherwise (sessions renew themselves while you use the site).

Settings

Account & usage → Settings holds the on/off switches for your account. Each is a single line; press and hold (or click) the ⓘ next to a switch for the full explanation.

  • Store history in the cloud (on by default) — keeps a copy of your conversations, files and search index in this site's storage so your history follows your account across devices. Conversations and files are stored encrypted; indexed material — large documents, project files and notes, project chats, and the search index itself — is stored readable, since retrieval needs readable text. Switching it off downloads everything to this browser and deletes the cloud copies. See "Chat history" above for how the encryption works.
  • Shodan host intelligence (off by default) — when on, if a question mentions an IP address or hostname, the site looks it up on Shodan and folds what it finds — open ports, running services, the hosting organization and ASN, location, and any known CVEs — into the research so the answer can use and cite it. Only the IP or hostname itself is ever sent to Shodan, never your question or anything about your account, and only when your message actually names a host. It works whether or not web search is on.
  • Google Maps & Street View (off by default) — when on, if a question names a street address (or you attach a photo carrying GPS location), the site looks it up on Google Maps Platform: it resolves the place with the Places API (name, address, type, rating), confirms Google Street View coverage and its imagery capture date, and pulls a road map — then folds those details and clickable Maps/Street View links into the research, and hands the Street View (several angles around the spot) and map images to a vision-capable model to describe. When Street View coverage exists you also get an inline, drag-to-navigate Street View right in the answer. Only the address itself (or a photo's coordinates) is ever sent to Google, never your whole question or anything about your account, and only when your message actually names an address. It works whether or not web search is on.

Privacy, in full

  • Conversations: processed by Berget.ai (EU) to generate answers. Saved locally in your browser, encrypted (see "Chat history" above) — erased only when you delete a conversation there or clear this browser's site data. Chats inside a project are the exception: they're indexed for retrieval and stored readable, like the project's documents (see "The exception" above).
  • The interaction log: to improve the service, the server logs each completed exchange — your question, the answer, the searches run and sources found, timings, and any error. For conversations that should never touch this server, the ghost button takes you to ghost mode — deepresearch.se/cure, where the server never receives your messages in the first place. (API clients can still send incognito: true per conversation to suppress the log row.)
  • Project vault copies: made only when you press Store on a project's "Encrypted copy, keyed by a secret". The archive is encrypted in your browser under a secret only you hold — the server stores unreadable bytes it has no key for (not even derivable), and switching cloud storage off never touches these copies. Load one back with 🔑 Load project from secret in the History panel.
  • Search queries: sent to Exa only while web search is ON, and only as short queries the AI derives from your question. Exa retains queries by default (zero data retention is an enterprise-only Exa feature we don't currently have) — see "Sensitive topics" above for the two-step workflow.
  • Shodan lookups: only if you turn on Shodan host intelligence in Settings (off by default). Then, when a message names an IP or hostname, that IP/hostname alone is sent to Shodan to fetch its public infrastructure record — never your question or anything identifying you.
  • Google Maps lookups: only if you turn on Google Maps & Street View in Settings (off by default). Then, when a message names a street address (or you attach a located photo), that address alone (or the photo's coordinates) is sent to Google Maps Platform to fetch the place, its Street View coverage and a road map — never your whole question or anything identifying you.
  • What our server keeps: your account (Google email, name), per-request usage numbers — token counts, search counts, durations — used for the quota bars, and the interaction log above (chats in ghost mode — se/cure — never reach the server to begin with).
  • Cookies: dr_session (keeps you signed in) and dr_privacy_ack (remembers you saw the privacy notice).
  • In your browser only: model choice, time target, and the web-search switch position (localStorage). If you switch away mid-research, a small marker (the conversation and request ids plus those same settings — never any message text) is also kept there briefly so the app can pick the finished answer back up when you return, even if the phone closed it in the background; it clears itself once the answer arrives, and expires within 15 minutes.
  • Sign-in: Google only. We receive your verified email and name from Google; we never see your Google password.
