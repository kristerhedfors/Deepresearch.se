Run a local browsing agent

← Back to the app

A local browsing agent is a small web-search service on your own machine. Once it runs, Se/cure routes every live web search through it, straight from your browser: your research queries reach the web from your computer, not from any hosted search provider — and never touch this site's server at all.

Quick start — Mac & Linux

Needs Node 18+ (brew install node on a Mac). One line, no dependencies:

curl -fsSL https://deepresearch.se/cure/local-search/agent.mjs -o agent.mjs && node agent.mjs

Then tell Se/cure about it — Settings → Web search service:

  1. Backend: Exa-compatible — your service
  2. Service URL: http://localhost:8099
  3. Flip the web knob on and ask a research question.

Prefer not to leave a file behind? Pipe it:

curl -fsSL https://deepresearch.se/cure/local-search/agent.mjs | node --input-type=module

Windows

Same script, PowerShell fetch (needs Node 18+):

iwr -useb https://deepresearch.se/cure/local-search/agent.mjs -OutFile agent.mjs; node agent.mjs

Pick how deep it browses

The default engine (browse) reads the result pages and hands your research real text excerpts, not just search snippets. Two alternatives:

Snippets only — fastest

Titles and search snippets, no page loads:

ENGINE=serp node agent.mjs

A real browser — JS-heavy sites

Renders each result page in headless Chromium, so sites that only work with JavaScript still yield text:

npm i playwright && npx playwright install chromium && ENGINE=playwright node agent.mjs

On Windows set the variable first: $env:ENGINE="playwright"; node agent.mjs

SearXNG — your own metasearch (Docker)

SearXNG aggregates Google, Bing, Brave, DuckDuckGo and more without tracking you. Start one with its JSON API enabled:

mkdir -p searxng && printf 'use_default_settings: true\nserver:\n  limiter: false\n  secret_key: "%s"\nsearch:\n  formats: [html, json]\n' "$(openssl rand -hex 16)" > searxng/settings.yml && docker run -d --name searxng -p 8888:8080 -v "$PWD/searxng:/etc/searxng" searxng/searxng

SearXNG doesn't send the CORS headers a browser-direct call needs, so run the agent in front of it as a thin proxy:

SEARXNG_URL=http://localhost:8888 node agent.mjs

Then use the same settings as the quick start (Exa-compatible, http://localhost:8099). If your SearXNG already sits behind a CORS-adding reverse proxy, you can instead pick backend SearXNG — your instance and point it there directly.

Options

All configuration is environment variables:

VariableDefaultWhat it does
PORT8099Listen port.
HOST127.0.0.1Bind address — localhost only unless you widen it.
ENGINEbrowseserp, browse, playwright or searxng (auto when SEARXNG_URL is set).
API_KEYnoneWhen set, requests must carry it (x-api-key). Paste the same key into the settings drawer. Recommended if you expose the agent beyond localhost.
SEARXNG_URLnoneYour SearXNG instance to proxy.
ALLOW_ORIGIN*CORS origin; tighten to https://deepresearch.se if you like.
PAGE_FETCH_CONCURRENCY3How many result pages load at once in browse/playwright.

Check it's alive

curl -s 'http://localhost:8099/search?q=test'

The service speaks the Exa-compatible wire (POST /search → {results:[{title,url,highlights}]}) plus GET /healthz, so anything else that speaks it drops in the same way.

Good to know

  • Privacy. With the agent configured, a search query goes browser → your machine → the web. deepresearch.se never sees it. The agent's default engines fetch results from DuckDuckGo, so that query (from your IP) is what the outside world observes; the SearXNG route spreads it across engines of your choosing.
  • Browsers. Chrome, Edge and Firefox allow an HTTPS page to call http://localhost; the agent also answers Chrome's private-network preflight. Safari blocks it as mixed content — use another browser, or put the agent behind a local HTTPS proxy.
  • Fail-soft. If the agent is down or unreachable, research degrades to the offline harvest (or a server grant if you hold one); the answer still arrives.
  • Keep it local. The agent binds 127.0.0.1 by default and logs nothing to disk. If you open it up to a LAN or the internet, set API_KEY and tighten ALLOW_ORIGIN.

More background on self-hosted search and the two-tier privacy model: the Se/cure documentation.
