How DeepResearch.Se/cure works

← Back to the app

This page documents the client-side tier, DeepResearch.Se/cure — the khaki app that runs in your browser with no account. Looking for the signed-in tier with hosted web search, cloud storage and accounts? Read the DeepResearch.Se/rver documentation.

What this tier is

Se/cure is a deep-research assistant that runs in your browser. You bring your own AI provider (an API key, or a model server on your own machine), and the research pipeline — planning, knowledge gathering, gap checking, synthesis, fact-checking — runs as client-side code on this page.

Privacy in one line: this server is in no data path. It serves the page's static files and public replay pages, and nothing else — your questions go directly from your browser to the AI provider you configure, and your chats, projects and API keys are stored only in this browser, encrypted. The server could not log your conversations or keys even in principle. (The two narrow, opt-in exceptions — borrowed web search and a borrowed research space — are described below, and each is disclosed in the app when active.)

Chats are stored in this browser only

Everything you make here — conversations, projects, notes, the search index, and the API keys you enter — is sealed into one encrypted archive (AES-256-GCM) kept in this browser's local storage, locked by a master secret that never leaves your device. There is no cloud copy: nothing is uploaded, nothing syncs, and no account exists that anything could sync to.

  • This browser is the only copy. Another device or another browser starts empty. To move or back up your work, use a project secret or a .drc backup file (below) — those are the only transports, and both stay encrypted end to end.
  • Clearing this browser's site data deletes your work for good. There is no server-side copy to restore from — that absence is the privacy guarantee itself. Export a backup first if it matters.
  • Your API keys live inside the same sealed archive — they are never sent to this server, and reach only the provider they belong to.

This browser-only storage is specific to Se/cure — the two tiers are opposites by design. On the signed-in Se/rver tier, history is encrypted in the browser and an encrypted copy is always stored in the cloud (there is no switch), so it follows your account across devices; see the Se/rver documentation for that tier's storage story.

Backups and moving devices

  • Project secret — saving your session as a project generates a master secret (the project form works with password managers, so 1Password or Apple Passwords can keep it). On any device, /my/project-… plus the secret reopens the project.
  • .drc backup file — the settings drawer can export your whole sealed archive as a downloadable file. Restoring it (file + secret) on any device brings everything back; importing never overwrites newer local work — the newer state wins and the other's chats merge in.

Bring your own AI

Open the settings drawer (the gear icon in the header) and paste an API key — the provider is detected from the key itself (sk-… OpenAI, gsk_… Groq, sk_ber_… Berget). Your browser then calls that provider directly; the conversation travels from your device to your provider and nowhere else.

  • OpenAI, Groq, Berget — the providers whose APIs a browser can call directly. The key is yours, the usage is billed to you, and the provider's own data policies apply to what it receives.
  • Your own local server — point Se/cure at an OpenAI-compatible server on your own machine (Ollama, LM Studio, llama.cpp) by its base URL, no key needed. This is the strongest privacy mode: no third party receives the conversation at all.

You can type immediately — the first send without a key answers with a friendly pointer to the settings drawer, never an error wall.

Research without web search

The spiderweb knob works here too: with research mode on, the pipeline plans angles, gathers what the model actually knows (and is honest about the limits of that knowledge), audits the gaps, runs a follow-up round, writes the answer on your chosen model, and fact-checks it. By default this is completely offline research — no web request is made beyond the calls to your own provider.

The time slider stays active regardless of the knob: depth is how deep the assistant goes over whatever sources are available, so even with the knob off it still shapes the answer — the model's reply scales from a quick brief up to a full structured report as you raise the target.

Borrowed capabilities — the two disclosed exceptions

Fresh web results normally require the Se/rver tier. Two opt-in, quota-metered, time-limited mechanisms can lend a Se/cure session bounded server capabilities without an account. Both are off unless you (or a link you follow) bring one, both show up in the app when active, and a master switch in settings turns them off entirely.

  • Temporary web search — a signed-in user crossing over (the ghost), or a shared …/cure?ws=… link, carries a grant for a fixed number of live web searches routed through this server's search key. Only the short search query reaches the server and the search provider — never your conversation.
  • Secure research space — a shared bundle can also lend a proxied AI model. Be aware: using a borrowed model necessarily routes the conversation itself through this server to the model provider (an AI call carries the prompt) — the one case where a Se/cure session's content touches this server. It is clearly disclosed in the app (a connected-APIs banner and a settings row) whenever it is active.

Sharing and replays

  • Secure workspaces — seal a fully configured session (keys, settings, chats, any borrowed grants) into one offline link under a password. The workspace rides the link's #fragment, which browsers never send to any server.
  • Published replays — pages under /cure/<name> are frozen, read-only research sessions published by the site owner. Opening one seeds it as a conversation here, so continuing it on your own key is just typing.

Execution sandbox

The experimental execution sandbox exists on this tier too (settings drawer): for questions that call for running code, a small Linux environment boots inside this browser tab and the assistant runs shell commands in it. Nothing about the commands or their output leaves your device.

Privacy, in full

  • Conversations: travel only between your browser and the AI provider you configured (or stay entirely on your machine with a local model). This server never receives them — except through the borrowed research space's proxied model, if you knowingly use one (see above).
  • Storage: everything rests in this browser as one encrypted archive under your master secret — chats, projects, search index, API keys. No cloud copy, no account, no server-side log of your messages (there is nothing to log: the messages never arrive).
  • Web searches: none, unless a borrowed web-search grant is active — then only the short AI-derived query reaches the server and the search provider, metered against the grant's quota.
  • Cookies and sign-in: none on this tier. No account exists, and the page sets no identifying cookie.
  • The dimmed buttons: features that need a server — accounts, hosted web search, cloud documents, the camera — appear dimmed where the signed-in app has them, and explain themselves when tapped. They belong to DeepResearch.Se/rver, the hosted tier.
