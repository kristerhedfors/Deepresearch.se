Settings · this browser
Provider API keys

Your key goes from this browser straight to the provider — never through this site's server. With a saved project it's stored only inside your sealed blob, in this browser.

Local model server
Run the model YOURSELF and point Se/cure at it — any OpenAI-compatible server works: Ollama (http://localhost:11434/v1), LM Studio (http://localhost:1234/v1), llama.cpp (http://localhost:8080/v1). No API key, no account — and nothing leaves your device: every research phase runs on your own machine. Ollama must allow this page's origin: start it with OLLAMA_ORIGINS=https://deepresearch.se (LM Studio and llama.cpp allow cross-origin calls by default).
On-device model Experimental
Runs a 1-bit Bonsai model (a compressed Qwen3.6) inside this browser on your device's GPU — after the one-time download, research works with no network at all: nothing you type ever leaves this device, not even to a model provider. Each model is a large one-time download (shown before anything starts) stored only on this device, with a Delete button to free the space. Phones need WebGPU (Android Chrome works; iOS Safari is experimental) and answers stream slower than hosted models.

        

      


      
      

        

          Web search (via Se/rver)
            
          
          
        

        
A signed-in Se/rver session granted this
          browser a small, fixed number of live web searches routed through the server's search key — so your
          research can use fresh web results even while you run your own (e.g. local) model. Only the search
          query leaves this browser (never your conversation); the allowance is fixed for this session and
          metered on the server. Turn it off to stay fully client-side.

        
      


      
      

        

          Web search service
            
          
        

        
Point live web search at your OWN self-hosted service —
          a SearXNG instance or any Exa-compatible endpoint — called directly from this browser, so
          no query touches this site's server at all. Your service must send CORS headers
          (Access-Control-Allow-Origin). The URL and key are stored only inside your sealed project,
          in this browser. Leave it on “Server grant” to use a granted allowance instead (or nothing). Recipes for
          running your own service: the local-web-search guide.

        
        

          
          
          
        

        
      


      
      

        

          Secure research space
            
          
          
        

        
A signed-in Se/rver account LENT this
          browser a temporary, quota-limited research space: proxied access to its server-side keys, so you can
          research without your own keys. The web search half sends only a query; the LLM API half routes
          your conversation through the server on the account's model key (so, unlike the rest of Se/cure,
          the server DOES see those messages) — it's opt-in and bounded. Turn it off to go fully client-side again.

        
      


      
      

        

          Se/rver token
            
          
          
        

        
One ticket, one JWT: a single temporary token bundling this
          session's borrowed allowances — web search (only the query leaves this browser) and the
          LLM API (routes your conversation through the server to Berget, metered). It is named a
          Se/rver token so you never forget: using it sends data to a server. It opens
          upstream APIs only — it can never read any Se/rver data (no projects, no
          chats), and it is never a login. Quotas are metered server-side and can be adjusted or revoked by whoever
          minted it. Turn it off to stay fully client-side.

        
      


      
      

        

          Secure workspace
            
          
          
        

        
Package this session — API keys, settings, chats, even a borrowed
          web-search or research-space allowance — into one link that IS the workspace: everything rides
          encrypted in the link's #anchor, which browsers never send to any server. Whoever you give the link
          (and its password, shared separately) opens a fully configured Se/cure session,
          completely offline. Same mechanism as hacka.re's shared links.

        Your workspace lives at /cure/workspace — share it as one offline link.
      


      

        

          Execution sandbox (bash) Experimental
            
          
          
        

        
Boots a real Linux in this browser so the assistant can
          run shell commands to answer you (e.g. “run ls”). Everything runs on your device — nothing leaves the
          browser. The first use downloads a Linux image, so it's slow to start.

        
      


      

        

          Introspection
            
          
          
        

        
Ask about this site's own implementation (“how are you built?”,
          “show me src/pipeline.js”) and the assistant answers from a snapshot of the exact source code this
          deployment runs — fetched as a public static file, so the server still sees none of your conversation.
          The composer pane turns white titanium while it's on. With the sandbox also on, the whole source
          tree is mounted at /src in the Linux VM for real shell exploration.
